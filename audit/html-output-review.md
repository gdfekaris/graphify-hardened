# Browser-side review of generated `graph.html`

Recorded: 2026-05-01
Plan task: 2.4 — browser-side audit of generated HTML beyond the CDN script.
Subject: every code path in `graphify/export.py::to_html` and the helpers it
calls (`_html_styles`, `_html_script`, `_hyperedge_script`).

The output is intended to be opened directly off disk (`file://`) or served
locally. CSP enforcement under `file://` is partial across browsers; the goal
here is defence in depth, not a single hard wall.

## Inventory of HTML-emitting code paths

`to_html` writes one document built from these building blocks:

- `<!DOCTYPE html>` + static `<head>` — fixed strings.
- `<title>graphify - {title}</title>` — `title` is `output_path` run through
  `sanitize_label()` and `html.escape()`. Safe.
- `<script>{vis-network bundle}</script>` — vendored, integrity-pinned;
  bundle has zero `</script` substrings (verified at load via
  `_load_vis_network_js` guard) and zero `eval` / `new Function`.
- `<style>` block from `_html_styles()` — fully static, no interpolation.
- Sidebar shell (`#sidebar`) — fully static.
- Two `<button onclick="toggleAllCommunities(...)">` — static onclick handlers.
- `_html_script(nodes_json, edges_json, legend_json)` — inline `<script>` that
  builds vis datasets and DOM. All embedded JSON goes through `_js_safe`,
  which `json.dumps`-encodes and additionally replaces `</` with `<\/` so
  embedded data cannot terminate the script tag.
- `_hyperedge_script(hyperedges_json)` — inline `<script>` that draws hulls
  on the network canvas using `ctx.fillText` for labels. Canvas text is not
  HTML-interpreted.

## Findings

### F1. Anchor / `target="_blank"` rule — vacuously satisfied

The generated document contains **zero `<a>` elements**. All "navigation"
between nodes is implemented as `<span class="neighbor-link" onclick="...">`
elements that call `focusNode()` to re-center the vis-network canvas; nothing
opens a new tab or window. There is therefore no `target="_blank"` reverse-
tabnabbing surface today.

This is a property of the current code, not an enforced invariant. If a
future change introduces real anchors, it must carry `rel="noopener
noreferrer"` and pass any href through the helper added under F3.

### F2. URL schemes in href — vacuously satisfied today, but unsafe by default

No `href` attribute is ever emitted in the output. Node fields that *look*
like URLs — currently `source_file` (a filesystem path) — are rendered as
escaped plain text inside the info panel via the `esc()` helper, not as
hyperlinks.

This is also a property of the current code, not an enforced invariant. If
a future change exposes a node URL, source-file path, or any other
extracted string as a clickable link, the URL must be validated against an
allowlist of schemes (`http`, `https`, `file`) before being interpolated
into an `href` — `javascript:`, `data:`, and `vbscript:` URIs must be
rejected. To make this safe-by-default we add a `_safe_href()` helper in
`graphify/export.py` (see fixes below).

### F3. Missing `_safe_href()` helper

There is no reusable scheme-allowlist helper in `export.py`. We add
`_safe_href(value, *, allow_file=True)` returning a safe href string (or
`"#"` for rejected values). This is defensive infrastructure for future
href emission and is exercised by the regression test (Task 2.5) using
adversarial node-URL inputs.

### F4. Missing Content-Security-Policy meta tag

The generated `<head>` has no CSP. Add a `<meta http-equiv="Content-Security-Policy">`
tuned for the inline-script setup chosen in Task 2.1:

```
default-src 'self' 'unsafe-inline'; script-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'none'; object-src 'none'; base-uri 'none'
```

Notes:
- `'unsafe-inline'` is required by `script-src` because the page emits
  inline `<script>` blocks and inline `onclick` handlers, and by `style-src`
  (inherited from `default-src`) because of the inline `<style>` block and
  inline `style="…"` attributes generated by JS.
- `'unsafe-eval'` is intentionally **not** allowed: the vendored
  vis-network bundle contains zero `eval`/`new Function` calls (verified by
  grep against `graphify/static/vis-network.min.js`), and the inline
  scripts emitted by `export.py` use only DOM and template-literal patterns.
- `connect-src 'none'` — neither vis-network nor the inline scripts make
  `fetch`/`XHR`/WebSocket calls, so no network egress is permitted from
  the rendered page.
- `img-src 'self' data:` — vis-network may use `data:` URLs for icons.
- `object-src 'none'` and `base-uri 'none'` — defensive; close off
  `<object>`/`<embed>` plugin loading and rebased relative URLs.

CSP under `file://` is enforced on a best-effort basis by modern browsers
and ignores origin-based directives, but the directives that filter by
scheme/keyword (script-src keywords, connect-src 'none', object-src 'none',
base-uri 'none') still apply.

### F5. `sanitize_label()` coverage — gaps on edges and hyperedges

`sanitize_label` is applied to node `label`, node `community_name`, node
`source_file`, the document `title`, and the legend `label`. Two gaps:

- Edge `relation` (used as both vis-network edge label and inside the edge
  tooltip `title`) and `confidence` are passed through unchanged. The HTML
  flow does run them through `html.escape` for the tooltip, but they are
  not first scrubbed of control characters. An `EXTRACTED…` style
  string from a hostile corpus would survive into the tooltip.
- Hyperedge `label` (used by `_hyperedge_script` in `ctx.fillText`) is
  passed straight from `G.graph["hyperedges"]`. Canvas text is not HTML-
  interpreted, but a control-char-laden label can still distort rendering.

Fix: route edge `relation` and `confidence` and hyperedge `label` through
`sanitize_label()` at build time in `to_html`.

### F6. JSON breakout via `</` — already mitigated

`_js_safe` (`export.py:470`) replaces `</` with `<\/` in every JSON blob
before interpolation into `<script>` blocks. This survives a hostile
node label of `</script><img onerror=...>`. No change needed.

## Verdict

After the fixes from F3, F4, and F5, the generated HTML will:

- carry a CSP that permits no external network egress and no `eval`;
- contain no external `<script src=>` (vendored under Task 2.3a);
- have zero hrefs whose value comes from extracted data, with a guard
  helper available for any future emission;
- run every user-controlled string through `sanitize_label` before
  embedding.

Fixes are applied in subsequent `security:` commits.
